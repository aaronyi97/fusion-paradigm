# Fusion 范式

把一个判断 / 设计问题，交给"不同家族"的模型先各自独立看，再综合。核心不是"多调几个模型"，是 **防止它变成多叫一个模型来点赞**。

## 什么时候用 / 什么时候别用

- **用**：方案设计、架构判断、重要文章定稿、审计、复盘、重大决策——需要多角度、错了代价大的场合。
- **别用**：写代码 / 跑执行、查简单事实、短翻译、日常小改——这些一个模型从头干最稳，多模型反而打架、慢、贵。

> **地基铁律**：没有"第一轮独立判断"，就不要叫 fusion。只把你的方案丢给另一个模型审，那是 reviewer，不是 fusion——第一轮不给主方案，是整个范式的地基。

## 第一次用：先配一次（向导）

第一次触发时，先问用户两件事，按回答给配置路径——

1. **你主要用什么工具？**（Claude Code / Cursor / Windsurf / 只有网页版 ChatGPT+Claude / 只有 API key）
2. **你能调用哪几个不同家族的模型？**（至少两个不同公司的，如 Claude + GPT、DeepSeek + Kimi）

按回答加载对应落地方案（只加载命中那一档，不全读）：

| 用户情况 | 加载 | 自动化程度 |
|---|---|---|
| 只有网页版 AI | `references/04-webui-manual-sop.md` | 零门槛·纯手动复制粘贴 |
| 有 API key | `references/03-api-script.md` + `scripts/fusion_api.py` | 脚本骨架·需补 SDK 调用 |
| Claude Code + Codex | `references/02-claude-code-codex.md` | 作者同款·半自动 |
| Cursor / Windsurf 等单工具 | `references/02-claude-code-codex.md` 的"单工具"段 | 手动多轮 + 综合模板 |

## 配好之后：怎么用

用户说"用 fusion 范式帮我看一下 [某方案 / 决策]"，按 `references/01-protocol.md` 的 7 步流程跑：

1. 判断这值不值得 fusion（不值就提醒"单模型就够"）
2. 构造上下文包——只给问题 / 约束 / 证据，**不塞主对话的结论**（防锚定）
3. 第一轮：让不同家模型独立出方案 / 反对（看不到主方案）
4. 第二轮：再给主方案，让它找漏洞
5. 主对话逐条处理分歧，产出分歧表（不许"综合来看"糊过去）
6. 把真正的取舍交给人拍
7. 记一句：这次 fusion 改变了什么判断

## 灵魂在哪

完整的 7 步流程、两轮 prompt 模板、角色分工、防玩具自检，全在 **`references/01-protocol.md`**——那是跨工具通用的核心，不管你用什么家伙什，那套流程不变。先读它，再读你那一档的落地文件。

## ⚠️ 一句话安全
fusion 会把你的问题 / 方案发给**多个外部模型**——涉密、客户数据、未公开信息先脱敏或别用。详见 README「安全」段。
